# NFLVERSE v2 â€” Design Document

Status: **draft, awaiting approval**.
Target: a coherent, declarative build pipeline that understands the full shape of nflverse data up front and produces a DB optimized for LLM-written SQL.
Scope: complete rewrite of the build code. Consumer-facing schema (table names, column names, FK edges) stays backward-compatible **or strictly expands**; the DB gets more complete and easier to query, not different.

---

## 1. What we learned from v1 (the honest autopsy)

v1 worked. It hit 100% player-hub stitching, declared 60 FKs, passed orphan sweeps, shipped. But it **accumulated** rather than was designed:

- **4 ID-cleanup functions**: `clean_id`, `clean_gsis_id`, `clean_id_series`, `clean_gsis_id_series` â€” two semantics (loose vs strict) crossed with two interfaces (scalar vs vector). Each added reactively.
- **5 update modes**: `year_partition`, `year_partition_upsert`, `full_replace`, `upsert`, `bulk_parquet`. Mode selection is per-table, per-reason â€” `players` is upsert because of FK-parent-UPDATE issues; `games` is bulk_parquet because of schema drift; `snap_counts` is year_partition because of year-sliced files. Five paths to do roughly the same job.
- **4 player-enrichment passes**: `_fetch_players` (pandas bridge merge) + `enrich_players_from_player_ids` (SQL bridge fallback) + `stub_players_for_config` (per-child stubs) + `stub_players_from_child` (the actual stubbing) + `_preflight_id_merges` (pandas pre-insert name-match) + `recover_gsis_by_name` (per-child name-match at insert time). Each one solved a real problem; together they're archaeology.
- **Post-hoc augmentation** (`compute_missing_season_stats`, `backfill_season_stats_team`) bolted onto the end of the build because we kept finding edge cases.
- **Source discovery was manual**: we missed `stats_player_post_{year}.parquet` for months because nobody systematically diffed "what nflverse publishes" against "what download.py pulls." Same for `pfr_advstats/advstats_season_def.parquet`, `weekly_rosters`, `injuries`, etc.

The individual fixes are correct. The collection isn't coherent. v2 rebuilds the collection.

## 2. North Star principles

These drive every concrete design decision below.

### 2.1 LLM SQL ergonomics is the primary optimization target
Every design choice gets measured against: "can an LLM write a correct query without memorizing five quirks?" When ergonomics and source fidelity conflict (rare), ergonomics wins â€” but the conflict is declared, documented, and reversible.

### 2.2 Canonical GSIS is the lingua franca
Every player-bearing row in every table has `player_gsis_id` populated when a GSIS exists for that player â€” **regardless of whether the source file carried GSIS natively**. `snap_counts` (PFR-native), `qbr` (ESPN-native), `combine` (PFR-native), `pfr_advanced` (PFR-native) all get `player_gsis_id` backfilled from the `players` hub at build time. Consumer queries use GSIS everywhere; source-native IDs stay as cross-reference columns but aren't required for joins.

### 2.3 Maximum coverage: if a value is findable in another table, fill it
If `game_stats.team` is NULL for a row but `weekly_rosters` tells us what team the player was on that week, we fill it. If `season_stats` is missing a POST row but `game_stats` has the weekly rows, we aggregate. If `players.college` is NULL but `combine` or `draft_picks` knows the player's school, we fill it. Fill rules are **declarative and enumerable**, not scattered special cases.

### 2.4 Honest NULL for genuinely unknown values
We don't invent data. A value is NULL only when no source we have carries it. Pre-GSIS historical players keep NULL GSIS; practice-squad-only callups with no PFR page keep NULL PFR. The DB distinguishes "missing because source doesn't know" from "missing because we didn't bother to fill it."

### 2.5 Declarative over imperative
One config file describes the DB shape, the source mappings, the FK graph, the fill rules. The engine is uniform â€” same primitives for every table. Adding a new source is a config edit, not a code change.

### 2.6 Data shape is catalogued before we build
A discovery pass runs before the build, producing a manifest of every nflverse file available, its columns, its year span, its ID columns. The config references the manifest; the build fails loudly if a referenced file isn't in the manifest (and vice versa, optionally). We can't miss `stats_player_post_*` again.

### 2.7 The consumer-facing shape doesn't break
Table names, column names, FK edges that NFL_AI_AGENT relies on don't change except to add. New columns, new tables, richer coverage â€” yes. Rename `snap_counts` to `weekly_snaps` â€” no. Old queries keep working.

## 4. ID system design

### 4.1 Canonical ID per entity type

| Entity | Canonical | Alternates (cross-ref, optional on joins) |
|---|---|---|
| Player | `player_gsis_id` (`00-0033873`) | `player_pfr_id`, `player_espn_id`, `pff_id`, `otc_id`, `esb_id`, `nfl_id`, `smart_id` |
| Game | `game_id` (`2023_19_KC_BUF`) | `pfr_game_id`, `old_game_id`, `nflverse_game_id` |
| Team | `team` (3-letter: `KC`, `LAR`, ...) | none (nflverse normalizes historical teams, e.g. STLâ†’LA) |
| Official | `official_id` | `official_name` |

### 4.2 How `players` is built (the hub)

Declarative priority-merge, one pass, in pandas:

```
SOURCES, in priority order (later ones fill NULLs from earlier, never overwrite):
  1. players/players.parquet          (canonical master, every ID column)
  2. external/db_playerids.csv        (GSISâ†”PFRâ†”ESPNâ†”PFF bridge)
  3. weekly_rosters/roster_weekly_*   (adds ~680 fringe-player GSIS we currently miss)
  4. draft_picks/draft_picks.parquet  (adds pre-GSIS draft pick PFR ids, college, draft info)
  5. combine/combine.parquet          (adds prospect PFR ids that never made a roster)
  6. [child tables w/ name+pos info]  (last-resort stubs for PFR/ESPN-only players)
```

Each source produces one DataFrame keyed on `player_gsis_id`. Merged with a collision policy:
- If two sources disagree on a non-NULL value (e.g., `latest_team`), **later source wins** for temporal fields (team changes over time) and **earlier source wins** for biographical fields (birth_date, college, position). Policy declared per-column in the config.
- Duplicate GSIS rows in a single source â†’ dedup keeping first.
- Source-without-GSIS rows (PFR-only stubs) get name-match-resolved against GSIS-bearing rows before being emitted as independent stubs (the v1 preflight merge, generalized).

Result: `players` is the union of every player referenced anywhere, with every available ID populated, and biographical fields filled from the best source that knows.

### 4.3 How child tables get GSIS backfilled

After `players` is built, **every child table that's missing a `player_gsis_id` column gets one added**, populated via the hub:

```
For a child table like snap_counts (native: player_pfr_id only):
  ALTER TABLE snap_counts ADD COLUMN player_gsis_id VARCHAR
  UPDATE snap_counts SET player_gsis_id = (
    SELECT p.player_gsis_id FROM players p
    WHERE p.player_pfr_id = snap_counts.player_pfr_id
  )
```

Done **declaratively** in the config:

```python
TABLES["snap_counts"] = {
    "source_file": "snap_counts/snap_counts_{year}.parquet",
    "year_range": "auto",
    "natural_key": ["player_pfr_id", "season", "week", "team"],
    "id_backfill": {
        "player_gsis_id": {"via": "players.player_pfr_id = self.player_pfr_id"},
    },
    "foreign_keys": [
        {"column": "player_pfr_id", "references": "players.player_pfr_id"},
        {"column": "player_gsis_id", "references": "players.player_gsis_id"},
    ],
    ...
}
```

Same mechanism for every PFR-only or ESPN-only source: `combine`, `pfr_advanced`, `pfr_advanced_weekly`, `qbr`, `depth_charts_2025` (which has both but some rows are ESPN-only), `snap_counts`.

Where the lookup returns NULL (PFR ID exists but no matching GSIS in hub), that row's `player_gsis_id` stays NULL. Honest gap.

### 4.4 Name-match recovery as a first-class phase

After ID-based backfill completes, any child row still lacking `player_gsis_id` but with a populated name column gets a **name + position + season + team**-match attempt against `players`. This is the v1 `recover_gsis_by_name` generalized and made declarative:

```python
TABLES["draft_picks"] = {
    ...
    "name_match_recovery": {
        "target_column": "player_gsis_id",
        "name_column": "pfr_player_name",
        "safety_filters": ["position_compatible", "season_active"],
    },
}
```

Runs automatically as a named phase after ID backfill. Each table declares whether to participate.

### 4.5 Invariant: every populated ID column resolves

After build, for every declared FK edge:
```
SELECT COUNT(*) FROM child c
WHERE c.fk_col IS NOT NULL
  AND NOT EXISTS (SELECT 1 FROM parent p WHERE p.pk_col = c.fk_col)
```
must return 0. Verified in `validate.py`. This is just the v1 orphan sweep, formalized.

## 5. Cross-table fill rules

Beyond ID backfill (Â§4), v2 systematically fills missing **data** from sibling sources. Every rule is declared, enumerable, and explainable.

### 5.1 Fill categories

| Category | When it fires | Example |
|---|---|---|
| **Hub-derived** | Child has a column the hub can provide | `snap_counts.team` NULL â†’ from `players.latest_team` ... NO, bad example, team changes over time. Use: `snap_counts.player_gsis_id` NULL â†’ backfill via `players` |
| **Sibling-aggregated** | Row is missing in one table but derivable from finer-grained sibling | `season_stats` POST row missing â†’ aggregate `game_stats` weekly rows (v1's `compute_missing_season_stats`, now declarative) |
| **Temporal-neighbor** | Value exists in a neighboring week/season | `weekly_rosters` missing jersey for week 6 but has it for week 5 â†’ carry-forward |
| **Cross-source** | One source has the value, another doesn't | `players.college` NULL â†’ fill from `combine.school` or `draft_picks.college` |
| **Computed** | Can be calculated from components already present | `season_stats.passer_rating` for derived rows â†’ compute from components when filled by aggregation |

### 5.2 Declarative rule examples

```python
FILL_RULES = [
    {
        "name": "season_stats_post_from_game_stats",
        "target_table": "season_stats",
        "target_scope": "missing_rows",
        "source": "game_stats",
        "key": ["player_gsis_id", "season", "season_type"],
        "aggregation": {
            "sum_cols": [...],            # from v1's _SS_SUM_COLS
            "max_cols": ["fg_long"],
            "compute_from_components": {
                "passer_rating": "formula(completions, attempts, passing_yards, passing_tds, passing_interceptions)",
                "completion_percentage": "formula(completions, attempts)",
            },
        },
        "metadata_source": "players",  # player_display_name, position, etc.
    },
    {
        "name": "players_college_from_combine",
        "target_table": "players",
        "target_scope": "null_values",
        "target_column": "college",
        "source": "combine.school",
        "join": "players.player_pfr_id = combine.player_pfr_id",
    },
    {
        "name": "players_college_from_draft_picks",
        "target_table": "players",
        "target_scope": "null_values",
        "target_column": "college",
        "source": "draft_picks.college",
        "join": "players.player_gsis_id = draft_picks.player_gsis_id",
        "priority": "after_combine",  # combine wins when both have a value
    },
    ...
]
```

Engine applies rules in declared order, idempotently. Every rule produces a summary line:
```
fill rule: players_college_from_combine      filled 1,247 NULLs
fill rule: players_college_from_draft_picks  filled 3,891 NULLs
```

### 5.3 Ratio-column handling (season_stats specifically)

v1 left ratio columns NULL for derived POST rows; v2 **computes them from summed components** when the components are themselves present. A declarative `compute_from_components` rule captures each ratio's formula:

```python
"passer_rating": "((((completions/attempts)-0.3)*5) + ((passing_yards/attempts)-3)*0.25 + ...)",
"completion_percentage": "completions * 100.0 / NULLIF(attempts, 0)",
"fg_pct": "fg_made * 100.0 / NULLIF(fg_att, 0)",
...
```

Result: LLM queries see populated ratios on every row. Equivalent to the nflverse pre-aggregated feed's ratios, just computed by us when the source feed skips the player.

## 6. Build engine design

### 6.1 Phases (strict order)

```
Phase 0: Prepare
  - Load manifest (scripts/v2/catalog.py build or refresh)
  - Load config (validated against manifest: every declared source exists)
  - Delete or back up existing DB (user choice, flag-controlled)

Phase 1: Ingest raw â†’ staging
  - For each declared source: read parquet(s), apply renames, apply ID cleanup
  - Emit staging DataFrames in memory (or temp DuckDB tables for huge ones like PBP)
  - No FK enforcement yet â€” staging is dirty by design

Phase 2: Build players hub
  - Priority-merge sources per Â§4.2
  - Name-match resolution for PFR/ESPN-only stubs against GSIS-bearing rows
  - Emit final `players` DataFrame (still in memory)

Phase 3: Create DDL + load parents
  - Create `players`, `player_ids`, `games` tables with full FK-bearing DDL
  - Insert from staging

Phase 4: Load children in FK-dependency order
  - For each child table: create DDL, insert from staging
  - FK enforcement active â€” violations fail loudly

Phase 5: ID backfill
  - Apply every `id_backfill` rule from config
  - ALTER TABLE ADD COLUMN + UPDATE ... FROM players
  - Emit counts per rule

Phase 6: Name-match recovery
  - For each table declaring name_match_recovery, resolve remaining NULLs
  - Emit counts per table

Phase 7: Fill rules
  - Apply every rule from Â§5.2 in declared order
  - Emit counts per rule

Phase 8: Views
  - Create v_depth_charts and any other declared views

Phase 9: Indexes
  - CREATE INDEX per declared index in config

Phase 10: Validate
  - Orphan sweep across every declared FK
  - Row-count parity checks (vs manifest expectations)
  - Column-completeness report (how many NULLs per column per table)
  - Honest-gap report (rows where every source was consulted and nothing filled)

Phase 11: Mirror to SQLite (optional, flag-controlled)
```

### 6.2 Primitives (these are the only operations the engine does)

Five, total:

1. **`load_parquet_source(source_spec) -> DataFrame`**
   Reads one or more parquets per the source spec; applies renames; applies ID cleanup.

2. **`merge_hub(sources, config) -> DataFrame`**
   Priority-merge for building `players`. Only used once.

3. **`write_table(name, df, ddl_spec, conn)`**
   Generates CREATE TABLE DDL from the schema spec (columns + FKs + UNIQUEs); inserts the DataFrame.

4. **`apply_fill_rule(rule, conn) -> int`**
   Executes one declarative fill rule; returns rows filled.

5. **`validate(spec, conn) -> ValidationReport`**
   Runs all declared invariants; returns structured report.

Everything the current v1 does gets expressed in terms of these five. No more mode-based branching (`upsert` vs `full_replace` vs `year_partition` vs ...).

### 6.3 ID cleanup â€” one function

```python
def clean_id(series, kind="generic"):
    """Normalize ID columns. kind='gsis' enforces the GSIS regex; kind='generic'
    only strips junk sentinels. One function, two modes, called from one place
    in the engine â€” never scattered across _fetch_* functions."""
```

Applied uniformly at source-load time based on each source's declared `id_columns`. That's it.

### 6.4 DDL generation

From the config alone:

```python
TABLES["season_stats"] = {
    "columns": "auto_from_source_schema",  # union of all year files
    "force_types": {"game_id": "VARCHAR"},  # type drift override
    "unique": ["(player_gsis_id, season, season_type)"],
    "foreign_keys": [
        {"column": "player_gsis_id", "references": "players.player_gsis_id"},
    ],
    "indexes": [("player_gsis_id", "season")],
}
```

Engine inspects the staging DataFrame's dtypes, maps to DuckDB types, emits CREATE TABLE. No hand-written DDL anywhere.

## 7. Verification framework

Single `validate.py`. Runs after every build. Produces a structured report. Build fails (non-zero exit) if any **hard invariant** violates.

### 7.1 Hard invariants (must hold; build fails otherwise)

1. Every declared FK: 0 orphans.
2. Every declared UNIQUE: 0 duplicate values.
3. Every table declared in config: exists and has rows (unless declared `allow_empty`).
4. Every declared column: present in the corresponding table.
5. `game_stats` â†” `season_stats` completeness: 0 (player, season, type) combos in game_stats absent from season_stats.
6. `players` hub completeness: every GSIS referenced from any child's `player_gsis_id` column resolves.
7. SQLite mirror row-count parity (when SQLite target is enabled).

### 7.2 Soft invariants (warnings; build completes)

1. Column NULL-rate: if any column jumps >5pp vs the prior build's NULL-rate, warn.
2. Table row count: if any table jumps >5% vs the prior build, warn.
3. Fill-rule productivity: if a rule fills 0 rows for the first time, warn (source data may have changed, or the rule is now moot).

### 7.3 Coverage report

Always produced:

```
Per-table coverage:
  players                 24,992 rows  |  gsis 98%  pfr 89%  espn 65%  birth_date 94%
  season_stats            61,589 rows  |  gsis 98%  team 100%  passer_rating 87% (87% is non-QBs)
  snap_counts            324,611 rows  |  gsis 99.2%  pfr 100%  (0.8% pre-GSIS)
  ...

Per-player-ID fill summary:
  Hubâ†’child GSIS backfill: +28,431 GSIS values filled into PFR-native tables
  Name-match recovery:     +8,247 additional GSIS values filled
  Honest-gap total:        1,823 rows where no source could resolve GSIS (1999-2002 mostly)
```

Prints on every build. Easy to diff build-over-build.

### 8.3 Parity checks (v1 vs v2)

Before swap, all of these must hold:

- **Row counts**: every v1 table's row count â‰¤ v2's count (v2 can only add).
- **Column presence**: every v1 column exists in v2 with same name and compatible type.
- **FK count**: v2 â‰Ą v1 (v2 should declare more, not fewer, given new tables).
- **LLM canary queries**: 10-15 queries we define upfront (e.g., "top 2023 POST rushers", "2024 LB snap leaders with tackles", "Patrick Mahomes career passing stats by season") â€” v1 and v2 return identical or strictly-more-complete results.
- **NULL-rate regression check**: for every v1 column, v2's NULL rate is â‰¤ v1's (we filled more gaps, not introduced new ones).

If all pass, atomic swap. If any fails, the failure is the thing to fix before swap.

## 10. Open questions for your review

Before I start coding, a few design decisions that could go either way. I have defaults below; flag any you want changed.

1. **Natural-key UNIQUE vs PRIMARY KEY on fact tables.** Current v1 uses UNIQUE on `(player_gsis_id, season, week)` etc. Default: keep that; PRIMARY KEY implies NOT NULL which we can't guarantee for pre-GSIS rows.
2. **Should derived `season_stats` rows be marked?** E.g., an `is_derived` column indicating "this row was computed from game_stats, not from nflverse's pre-aggregated feed." Current v1: no flag, same treatment. Default: still no flag â€” LLMs shouldn't need to care; computed values are authoritative. If you want auditability later, we add it.
3. **How aggressive should name-match recovery be?** Default: run only when no ID-based lookup succeeded, require name + position-compatible + season-active match, reject ambiguous. Don't fuzzy-match names (no "Mike Williams" vs "Michael Williams"). Conservative wins over aggressive.
4. **`qbr` source swap â€” replace or coexist?** Default: replace. nflverse's parquet is fresher (covers 2024-2025), has more players, matches the rest of our pipeline. The espnscrapeR CSV goes away.
5. **Rosters (annual, 1920-) and players_components â€” include?** Default: skip for now, hook in config so we can add later. The historical roster coverage mostly predates the modern stat tables and would bloat `players` with 1000s of pre-GSIS entries without corresponding stats to join.
6. **Weekly rosters as its own table, vs merged into a wider players table?** Default: own table (`weekly_rosters`). Players should be a slowly-changing master; weekly status (team changes mid-season, jersey changes, status) is week-granular and belongs in a separate table. Players table carries only the "latest/primary" values.
7. **Should v2 support running against nflreadpy (network source) as a fallback?** v1 had `build_db_nflreadpy.py` for this. Default: no. nflverse releases are stable; local parquet is the only source. If network-source support is needed later, it's a manifest adapter, not a fork of the build.
8. **FK enforcement mode in DuckDB.** Default: declared but honored on INSERT only (DuckDB's default). Consider `PRAGMA enforce_foreign_keys = true` globally? It's default-on in DuckDB â‰Ą1.0, so nothing to configure.

## 11. What v2 is NOT

- **Not a column renaming pass.** Existing column names stay (we agreed in v1 that nflverse-native column names, with the exception of `player_gsis_id`/`player_pfr_id`/`player_espn_id`, were fine).
- **Not a schema normalization.** No star-schema refactor, no splitting `game_stats` into per-position tables. The current wide format is correct for LLM SQL ergonomics (one `SELECT * FROM game_stats WHERE ...` answers most questions).
- **Not a consumer-facing API change.** NFL_AI_AGENT should not need a single query rewrite.
- **Not a data source replacement.** Still nflverse + dynastyprocess bridge. Not scraping PFR directly; not hitting ESPN APIs.

## 12. Success criteria

After v2 ships, the following are simultaneously true:

1. One config file (`scripts/v2/config.py`) is the single source of truth for the DB's shape. Adding a new table is a config entry, not a code change.
2. One `clean_id` function, two usage modes, called from one place.
3. 22 tables (vs 13), 80+ FKs (vs 60), with all audit-found gaps closed.
4. Every player-bearing row in every table has `player_gsis_id` when a GSIS exists for that player.
5. Every LLM canary query from Â§8.3 returns results at least as complete as v1.
6. The phase-10 coverage report runs automatically; no column has higher NULL-rate than v1.
7. Total build time on an M-series laptop or comparable: under 5 minutes for DuckDB (vs v1's ~2 min for main + ~1 min for PBP). Adding ~10 tables should not slow the build proportionally because the engine is uniform.
8. INGESTION.md fits in one screen of contents. Current is 12 sections and growing; v2 target is 6-8.

If any of these isn't true at ship time, we don't ship.
